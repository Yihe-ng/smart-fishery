# 智慧渔业项目技术介绍与开发文档

## 1. 项目概述

本仓库是一个前后端分离的智慧渔业系统，围绕养殖场景提供水质监测、病害识别、生长监测、投喂管理、设备状态、告警与系统管理能力。

- 前端目录：`frontend`
- 后端目录：`backend`
- 根目录职责：作为前后端并列工作区，同时保留少量根级说明文件

当前项目更适合描述为"可运行的业务中后台原型 + 部分真实工程能力 + 部分 mock 接口/数据并存"的状态：

- 前端主体功能、页面结构、路由体系、状态管理、样式体系已经成型
- 后端 API、WebSocket、数据库会话、算法入口已铺好
- 部分后端接口仍以 mock 数据返回，前端也保留了 mock 数据驱动页面的做法

## 8. 如何启动项目

## 8.1 根目录一键启动

根目录提供了 `dev.py`，用于在一个控制台窗口内同时启动前后端开发服务，并汇总日志输出。

### 使用方式

```bash
python dev.py
```

### 脚本启动前会检查

- 当前 Python 解释器是否可用
- `uv`、`pnpm` 是否已安装且在 PATH 中可执行
- `backend/pyproject.toml`、`backend/app/main.py`、`frontend/package.json` 是否存在
- `backend/.venv` 是否存在
- `frontend/node_modules` 是否存在

### 依赖缺失时的提示

- 后端依赖缺失时，终端会提示先执行：`cd backend && uv sync`
- 前端依赖缺失时，终端会提示先执行：`cd frontend && pnpm install`
- 如果 `pnpm` 没有加入 PATH，脚本会直接报错并提示修复 PATH

### 运行行为

- 后端日志前缀为 `[backend]`
- 前端日志前缀为 `[frontend]`
- 任一进程异常退出时，脚本会结束另一侧进程
- 按 `Ctrl+C` 会统一关闭前后端进程

## 8.2 前端启动

用户已确认的正确流程如下。

### 第一步：安装 pnpm

如果本机未安装 `pnpm`，先安装：

```bash
npm install -g pnpm
```

### 第二步：安装依赖

```bash
cd frontend
pnpm install
```

### 第三步：开发启动

```bash
pnpm dev
```

默认使用 Vite，本地开发端口来自 `.env.development`：

```text

```

开发代理目标地址：

```text
http://127.0.0.1:8000
```

### 第四步：生产构建

```bash
pnpm build
```

### 第五步：本地预览

```bash
pnpm serve
```

后端统一使用 `uv`。

### 第一步：进入后端目录

### 第四步：访问接口文档

如果不使用 `dev.py`，建议按以下顺序手动启动：

1. 先启动后端 `backend`
2. 再启动前端 `frontend`

## 9. 如何开发

## 9.1 新增前端页面

推荐流程：

1. 在 `src/views` 下新增页面目录
2. 优先使用 `<script setup lang="ts">`
3. 如果有复用逻辑，放入 `src/hooks/core` 或更合适的 composable 目录
4. 如果有接口调用，在 `src/api/<domain>` 下封装
5. 如果需要加入菜单路由，在 `src/router/modules` 补充模块
6. 如果是阈值、颜色、图表元数据，优先写入 `src/config`
7. 先用 mock 数据打通页面，再切真实接口

## 9.2 新增后端接口

推荐流程：

1. 在 `backend/app/api/v1/endpoints` 新增或扩展接口文件
2. 在 `backend/app/api/v1/api.py` 注册路由
3. 若涉及数据结构，补充 `schemas`
4. 若涉及数据库，补充 `models`、`crud`、`db`
5. 若涉及业务逻辑，沉淀到 `services`
6. 若涉及实时通信，补充 `websocket`

## 9.3 前后端联调方式

当前项目前端默认通过 Vite 代理转发 `/api` 请求到：

```text
http://127.0.0.1:8000
```

因此前端接口通常可直接写：

```ts
url: '/api/xxx'
```

不需要在本地开发时手动写死完整域名。

## 9.4 mock 转真实接口的建议

当前仓库大量页面采用 mock 优先模式，建议联调时遵循：

1. 先保留页面结构与交互
2. 用真实接口逐个替换 mock 数据
3. 保持返回结构与前端类型定义一致
4. 将阈值、状态映射、颜色映射保留在统一配置层，避免散落到页面里

## 10. 项目开发规范

## 10.1 前端规范

结合仓库代码和本地 skills，总结为以下规则：

- 默认使用 Vue 3 Composition API
- 默认使用 `<script setup lang="ts">`
- 页面、组件、hooks、config、api 分层清晰
- 复杂逻辑优先抽 composable，不把页面写成巨型 SFC
- 路由、菜单、权限、标签页交由现有框架接管，不要手写绕过
- 样式优先复用现有主题变量和通用类
- 图表颜色、指标顺序、阈值配置优先放在 `config`
